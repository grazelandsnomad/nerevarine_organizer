# 0.6 release notes

_In development._ Changes land here as they're committed; this file becomes the
GitHub release body when `v0.6` is tagged (release.yml reads it by version).

## New features

- **BAIN archive installer.** Mods packaged the Wrye Bash way - content split
  into numbered folders like `00 Core`, `01 Optional Textures`, `10 Alternate
  Meshes` - now get a package picker at install time instead of installing as
  one blob. The picker opens on a one-click choice - **Install everything** (the
  usual answer, and exactly what an install-all mod like Tamriel Rebuilt wants)
  or **Choose packages…** to tick a subset - and the chosen packages are merged
  into the mod. Re-installing or updating a BAIN mod remembers the packages you
  picked last time and pre-ticks them. FOMOD installers still take precedence;
  this only kicks in for the numbered-folder layout.

- **Merge optional downloads into an installed mod.** Some mod pages ship
  optional files that are meant to override the main download (OAAB Data's
  optionals, for example). When you download an optional file for a mod you
  already have installed, the "already installed" prompt now offers a third
  choice - **Merge into existing** - alongside Replace and Install separately.
  Merge overlays the new files on top of the existing mod folder, letting them
  override the current ones (last writer wins), so the override lands in the
  same mod with no second entry to juggle in the list - the equivalent of
  Mod Organizer 2's merge. Replace (treat as an update) and Install separately
  (a different file from the same page) behave as before.

- **Share a mod between modlist profiles (no second copy on disk).** When you
  run more than one modlist profile for a game - say a "main" Morrowind list and
  a "StarWind" total conversion - right-click an installed mod and pick **Share
  with profile…** to add it to another profile pointing at the *same* installed
  folder. No files are duplicated. You choose whether to copy your current
  configuration (enabled state, FOMOD choices, notes) or start it disabled at
  defaults, and from there each profile's settings, load order and toggles
  diverge independently. Shared mods are tagged in their tooltip. Removing or
  uninstalling a shared mod from one profile keeps the files on disk while any
  other profile still uses them; the files are only deleted when the last
  profile lets go. And if you reinstall, update, re-run a FOMOD wizard on, or
  merge into a shared mod in one profile, it automatically forks a private copy
  for that profile first - so your change lands only where you made it and the
  other profiles keep their version untouched.

- **Share or back up a modlist as a single file.** **Export Modlist** now
  writes the list in the manager's own lossless format - load order,
  enabled/disabled state, FOMOD and BAIN package choices, Nexus links, your
  notes and separators all included - and **Import Modlist** reads it back
  faithfully. Paths are remapped on import and mods are matched by their Nexus
  link, so the same file works as a personal backup, as a way to move a setup
  between machines, or to hand your whole load order to someone else - anything
  they don't have yet imports as a placeholder to download or point at. (The
  previous export only captured names, paths and links; older exports still
  import.)

## Bug fixes

- **A failed extraction no longer wrongly tells you to install 7z.** When a
  downloaded mod couldn't be unpacked, the error always ended with "Make sure 7z
  (p7zip-full) is installed" - even when 7z *was* installed and had clearly run
  (it exited with an error code, which only happens if it ran). The message now
  reflects the real cause: an extractor error code points at a corrupt or
  not-fully-downloaded archive and prompts you to re-download, and the "install
  it" hint only appears when the extractor genuinely can't be found.

- **Mods uploaded as a loose plugin now install.** Some mods (often small OpenMW
  ones) are uploaded as a bare `.omwaddon`/`.esp` file instead of inside a
  `.zip`/`.7z`. The installer always tried to *extract* every download, so those
  failed with a "7z fatal error". Downloads are now checked by content: a real
  archive is extracted as before, and a loose file is placed directly as a mod
  (named after the mod when the download itself carries no filename).

- **Downloads that arrive without a filename extension now install.** A few Nexus
  files come down named only as a bare ID with no `.zip`/`.7z` on the end, and
  installing one of those failed *every* time with a "7z fatal error" that looked
  exactly like a corrupt download - re-downloading never helped. The file was
  actually fine: the manager unpacks each download into a folder named after it,
  and with no extension that folder name collided with the downloaded file
  itself, so the unpacker couldn't create its output folder. The manager now
  always unpacks into a distinct folder, and these mods install normally. (Such a
  mod also used to land in the list under its raw download ID - e.g.
  `4c9017a6-…` - and now resolves to its proper Nexus title.)

- **Genuinely broken downloads are caught before the confusing extractor error.**
  When a download really is corrupt or incomplete, you used to get the same
  opaque "7z fatal error" at install time. Downloads are now sanity-checked as
  they finish - a truncated transfer, a server error page returned in place of
  the file, or an archive that fails an integrity test is detected, retried once
  automatically, and if it still won't come down cleanly the bad copy is kept
  (as `<name>.corrupt`) with a line written to a `download_diag.log` so the cause
  can be pinned instead of guessed. The clear next step - re-download, or grab it
  from Nexus and drag it onto the window - is shown instead of a cryptic code.

- **The window no longer freezes after installing or updating a mod.** Finishing
  an install/update used to lock the app up for a few seconds (the window greyed
  out, then recovered) - worst on a mods folder that lives on an NTFS/exFAT drive,
  where deleting the previous version's files is slow. The slow folder deletes now
  happen in the background instead of blocking the window: the old folder is
  vacated instantly and its files are cleared afterwards, so the list stays
  responsive. The mod's config (`openmw.cfg`/load order) is written exactly as
  before - only the cleanup moved off the main thread.

- **Flatpak LOOT now works.** When LOOT is installed as a Flatpak, "Sort with
  LOOT" used to run it but it detected no plugins and closed - the Flatpak
  sandbox couldn't see your mods folders or the OpenMW config. The manager now
  launches Flatpak LOOT through `flatpak run` with the filesystem access it
  needs (mods/plugins on any drive, plus read/write to `openmw.cfg`), so sorting
  works the same as a native LOOT install. Native installs are unaffected.

- **Custom separator text colour is kept in dark mode.** 0.5's dark-mode
  separator theming darkened each section's background but also replaced your
  chosen *text* colour with a fixed light grey, so a separator you'd recoloured
  came back looking different after the update. Dark mode now keeps your text
  colour whenever it still reads against the darkened background, and only
  substitutes a light label when your colour would genuinely be unreadable
  (dark text authored for a pale background). Light mode was never affected, and
  no stored colours are changed - this is purely how they're drawn.

- **Separators no longer swallow a neighbouring section, and you get a say in
  where they land.** "Add Separator" used to insert just after the last-clicked
  row - and if that row was a *collapsed* separator, the new one dropped in
  among its hidden mods and silently adopted that whole section. Now "Add
  Separator" (Mods menu or right-click on empty space) always appends a fresh,
  empty separator at the **end** of the list, and right-clicking a row offers
  **Add Separator Above** to place one precisely above that mod. Drag it wherever
  you like afterwards as before.

- **Toolbar text stays readable on a dark desktop theme.** The Profile
  label/picker and the light/dark toggle button picked their text colour from
  the app's own light/dark flag. But "light mode" restores your desktop's
  native palette - which on a dark theme (e.g. Breeze Dark) is itself dark - so
  the app ended up painting near-black text on a dark toolbar. The text colour
  now follows the *actual* background, so those controls read correctly whether
  the underlying palette is light or dark.

## Experimental

- **Mod deployment for Bethesda games (OG Oblivion).** Early, opt-in support for
  managing a non-OpenMW game: a profile for OG Oblivion now has **Mods → "Deploy
  to game"** and **"Remove deployed mods"**. Deploy links your enabled mods into
  the game's `Data/` folder (hardlink/symlink/copy), writes `Plugins.txt`, sets
  the load order by file timestamp the way Oblivion reads it, and configures
  `Oblivion.ini` so BSAs and loose retextures load. Everything is reversible -
  displaced game files are backed up and restored on "Remove deployed mods" - and
  re-syncs automatically before launch once you've deployed. **This is unverified
  on real hardware and gated behind an Oblivion profile; it does not affect
  OpenMW/Morrowind.** Design notes + a field-test checklist live in
  [`docs/design/bethesda-deployment.md`](../design/bethesda-deployment.md).

## Migration notes

- No state-format changes. Existing modlists, load orders, profiles and the
  light/dark theme choice carry over unchanged.
- The BAIN picker only appears for archives whose top level is entirely
  numbered packages and that aren't FOMOD installers; everything else installs
  exactly as before.
