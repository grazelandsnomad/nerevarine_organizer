# 0.3.1 release notes

Hotfix 1 on top of 0.3. Three user-reported issues.

## Bug fixes

### Completionist Patch Hub: empty `scripts/` folder, orphan `.omwscripts`

Installing Completionist Patch Hub (Nexus mod 58523) left all 17
`Completionist - <X>.omwscripts` files at the mod root and an empty
`scripts/` directory next to them. Each `.omwscripts` declared a script
under `scripts/Completionist - <X>/quests_<X>.lua` that wasn't actually
on disk - so OpenMW would fail to load them.

Two changes in the FOMOD wizard:

- **Patch-hub auto-tick.** A new "Pass D" in `FomodWizard::buildUi`
  detects `SelectAny` groups where every plugin's `<files>` includes
  an `.omwscripts` entry, and ticks them all by default. The previous
  defaulting passes only ticked options whose target mod was already
  in the user's list; with no landmass mods present, nothing got
  selected and the patch lua content never installed. The user can
  still untick any option before clicking Install.

- **No more empty `scripts/` placeholder.** `fomod_copy::copyContents`
  / `copyDir` now skip creating the destination folder when the source
  is empty. The Patch Hub's archive root contained an empty `scripts/`
  placeholder dir that the wizard used to materialize verbatim - now it
  doesn't appear if no real content lands in it. Helpers extracted
  from `FomodWizard` into a Qt-Core-only namespace so the rule is
  unit-testable; covered by 7 cases in `tests/test_fomod_copy.cpp`,
  including the Patch-Hub shape.

### "Mod Manager Download" overwrites a sibling file from the same mod page

Many Nexus mod pages bundle complementary optional files - e.g.
mod 58704 ships both *The Wretched And The Weird* (main file) and
*Sage's Backgrounds* (optional). The handler reused the existing
installed row when a new `nxm://` URL came in for the same `modId`,
stashed the prior install path in `PrevModPath`, and then
`addModFromPath` deleted it after the new download landed - silently
overwriting the first mod with the second.

The "already installed" prompt is now a three-way dispatch:

- **Replace existing** - the new download is an update of the existing
  mod; the old folder is removed once the new one lands.
- **Install separately** - the new download is a different file from
  the same mod page; lands in its own folder beside the existing entry.
- **Cancel** - abort the install.

Default focus is on Install separately - it's the non-destructive
choice and the more common case for mod pages that bundle
complementary content. Updates triggered via Nexus's "Update" button
or the in-app update flow are unchanged.

### Fallout: New Vegas pinned entry shipped half-done

The pinned-game dropdown entry for FNV was visible in 0.3 but the
detection / per-game install paths weren't tested. Commented out for
0.3.1; one-line uncomment to re-enable once the work resumes. The
Lutris exe finder and the multi-game profile loader stay in.
