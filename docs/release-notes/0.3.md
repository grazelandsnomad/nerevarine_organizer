# 0.3 release notes

## Bug fixes

### LOOT exits silently with "QIBusPlatformInputContext: invalid portal bus"

Under the AppImage build, the AppRun wrapper exports `LD_LIBRARY_PATH`,
`QT_PLUGIN_PATH` etc. so the bundled Qt6 can find its plugins. `QProcess`
inherits the parent env by default, so LOOT (also Qt) was loading our
plugins instead of its own and bailing out. Foreign Qt children are now
launched with the `*_ORIG` values that `linuxdeploy` stashes for this.
No-op outside AppImage.

### `forbidden_mods.txt` / `modlist_*.txt` saved one directory above binary

The path resolver had a legacy "one directory up" fallback from an old
in-tree layout (binary under `bin/Release_Linux/`, state files in
`bin/`). Once a state file existed there it was sticky.

Now treated as a one-time migration source: leftover state files there
get moved next to the binary on first run. New installs write next to
the binary; `AppDataLocation` as a fallback when not writable. AppImage
builds keep using `AppDataLocation`.

### OpenMW Launcher Content List wiped after Import / Remove-keep-on-disk

`syncOpenMWConfig`'s orphan scrub dropped any `data=` line under the
mods dir not claimed by a managed mod. After Import or Remove (without
delete-from-disk), previously-managed mods became unclaimed but their
folders were still on disk with plugins inside, and the Launcher
silently lost everything they contributed.

Sync now performs an orphan-managed rescue before the scrub: any
inside-managed `data=` whose folder still has plugins on disk is
re-emitted outside the managed `BEGIN/END` block. Folders with no
plugins still drop normally.

### Single-folder archives extract to the wrong data root

The post-extraction "where's the real mod root?" rule used a naive "if
exactly one top-level subdir, dive into it". This misfired for archives
whose top level was a bare asset folder (`meshes/...`) or had a single
subfolder plus loose `.esp` files alongside.

New helper `install_layout::diveTarget` (`src/install_layout.cpp`):
dive only when the single subdir is *not* itself an OpenMW asset
folder (`meshes`, `textures`, `sound`, ..., `fomod`) and there are no
top-level files alongside. Covered by 22 tests in
`tests/test_install_layout.cpp`.

### Mod removal leaves orphaned wrapper directories

Removing a mod with delete-from-disk left an empty wrapper dir behind
when the mod had been extracted from a single-subfolder archive. The
remove path now sweeps up empty parents between `ModPath` and
`m_modsDir`, stopping at any directory that still holds files.

### Launcher Content List wiped on close

OpenMW Launcher writes the user's selected paths to both `openmw.cfg`
and `launcher.cfg`. Vanilla `data=` entries set up via `openmw-launcher`
or `/etc/openmw/openmw.cfg` could legitimately live only in
`launcher.cfg`. Our orphan scrub then dropped them.

`syncOpenMWConfig` now reads `launcher.cfg` upfront and synthesises any
`data=` / `content=` not in `openmw.cfg` (and not under `m_modsDir`,
not provided by any managed mod) as external lines. The new
`openmw::readLauncherCfgDataPaths` helper has 5 added cases in
`tests/test_launcher_cfg_writer.cpp` (30 total).

### Importing a modlist didn't persist immediately

**File → Import Modlist...** only updated the in-memory list; imported
enable flags didn't reach `openmw.cfg` / `launcher.cfg` until the next
event triggered a save. Now `doImportNerevarineModList` finishes by
calling `saveModList`.

### NXM handler registration didn't survive AppImage relaunches

The KDE/KIO `.protocol` files are now written with `$APPIMAGE` (stable
on-disk path) instead of `applicationFilePath()`, which under AppImage
resolves to the ephemeral `/tmp/.mount_<random>/...` mount.

### Empty modlist save on startup

A new `m_modListLoaded` flag blocks `saveModList()` from writing until
the initial `loadModList()` has populated `m_modList`. A crash during
load could otherwise wipe the on-disk list.

### AppImage settings file wasn't writable

`nerevarine_prefs.ini` now goes to `AppDataLocation`
(`~/.local/share/nerevarine_organizer/`) under AppImage. Previously the
app looked for it next to `applicationDirPath()`, which is the
read-only FUSE squashfs mount.

### Header columns drift after maximise/restore and on tiling WMs

The header now reads `m_modList->viewport()->geometry()` directly
instead of computing from the scrollbar width. `changeEvent` defers
realignment one tick (Qt finishes layout after the event fires);
`resizeEvent` covers tiling WM snap gestures.

### Video review URL via mod-edit dialog wasn't shown

The video-review column delegate only queried the hardcoded lookup
table in `video_reviews.h`. Both paint and click paths now check
`ModRole::VideoUrl` first.

## Improvements

### MainWindow split into 23 modules

`src/mainwindow.cpp` shrank from 11,451 to 9,209 lines (-20%). No
behaviour change. Modules: UndoStack, ZoomController, FilterBar,
NotifyBanner, ColumnHeader, ForbiddenModsRegistry, GameProfileRegistry,
ToolbarCustomization, ScanCoordinator, BackupManager, BulkInstallQueue,
ReviewUpdatesDialog, launch_warnings, modlist_sort, send_to_dialog;
plus four anonymous namespaces promoted to named TUs (IniDoc,
nexus_name, conflict_scan, reboot_check).

### Faster mod removal

Pressing Delete no longer blocks the UI while scanning for redistributed
plugins.

- `collectDataFolders` is now cached per mod path. Invalidated on
  install / remove.
- The O(n) "is this plugin bundled in another mod?" pass runs on a
  background thread after deletion. UI returns immediately; if found,
  the prompt appears once the worker finishes.

Most noticeable under AppImage (FUSE mount), but applies everywhere.

### Log triage auto-opens after an OpenMW crash

OpenMW is now started as a monitored `QProcess` instead of
`startDetached`. On non-zero exit, the Log Triage dialog opens after a
deferred singleshot. Launcher-button launches stay detached.

### Bulk-install drip-feed

Right-clicking many not-installed placeholders and choosing Install
selected no longer fires all Nexus API requests in a tight loop. Mods
queue in `m_bulkInstallQueue` and a 1.2s timer pops one per tick.
Status-bar message shows the queue depth.

### NXM download reuses placeholder in-place

When an `nxm://` download starts, the handler scans for an existing
not-installed entry whose Nexus URL matches the same game + mod ID. If
found, that row is promoted to "installing" in place instead of a new
row appended at the bottom.

### Structured logging by subsystem

Log lines now carry a `QLoggingCategory` tag. Verbosity scoped per
subsystem via `QT_LOGGING_RULES`:

```sh
QT_LOGGING_RULES="nerev.install.debug=true;nerev.scan.debug=false" \
  ./NerevarineOrganizer-x86_64.AppImage
```

Categories: app lifecycle, install pipeline, Nexus API, openmw.cfg
sync, plugin load order, filesystem scans, modlist load/save, launch
flow, FOMOD, misc UI. Existing `qDebug` / `qWarning` calls still work.

The on-disk log gains numbered rotation: `log.txt` -> `log.txt.1` ->
`.2` -> `.3` -> `.4` instead of being deleted-and-restarted. Up to ~20
MB of history survives a crash. **Settings → Open Log Folder...**
opens the directory in the file manager.

### AppImage: dialog icons pre-warmed at startup

`QStyle::standardPixmap` and `QIcon::fromTheme` go through FUSE on
first access. Both are now pre-warmed at idle via
`QTimer::singleShot(0)` so the cost is invisible.

## Tests

### Property/fuzz tests for `openmw.cfg` synthesis

`test_openmw_config_writer_fuzz` runs 500 deterministic random
scenarios against `openmw::renderOpenMWConfig`, checking nine
invariants:

- Exactly one BEGIN/END pair, in order.
- No duplicate `content=` / `groundcover=` filenames.
- Suppressed plugins absent from both.
- Disabled / uninstalled mods don't leak into `content=`.
- Load order respected.
- Every enabled+installed plugin dir becomes a managed `data=`.
- `render(render(input))` is byte-identical to `render(input)`.
- Suppressed plugin filenames don't sneak into managed `data=` paths.

Failures dump seed + loadOrder + offending output. Reproduce a single
case with `./test_openmw_config_writer_fuzz --seed=N`.

## CI

Tests now run on every push/PR via `.github/workflows/test.yml`. Same
Qt 6.7.3 + GCC 14 setup as the AppImage build, but skips
linuxdeploy / icon-resize / Qt-deploy steps for fast PR feedback.

The AppImage workflow picks up two defensive patches: `imagemagick`'s
`policy.xml` is relaxed for PNG (the `convert PNG` call was failing
since CVE-2016-3714 changed the default), and the build step sets
`CC=gcc-14 CXX=g++-14` explicitly so cmake can't fall back to the
runner's older default.
