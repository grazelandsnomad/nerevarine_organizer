# 0.4 release notes

Two themes: **modlist profiles per game** (test a Wabbajack without
wiping your daily-driver setup) and **BSA-shipped mods now actually
work** (Authentic Signs IT and friends).

## New features

### Modlist profiles per game

Each game now holds N modlist profiles, every profile with its OWN mods
directory on disk - no sharing, no overwriting between profiles.

- **Toolbar picker.** New "Profile: <name> ▾" control sits next to the
  game button. The dropdown lists every profile for the active game,
  plus *Rename "<active>"…*, *New profile…*, *Manage profiles…*.
- **Manage profiles dialog.** Create / clone / rename / delete /
  set-active. Clone is "clone-empty" for the mods dir (the new profile
  picks its own location at first install) and "clone-full" for the
  modlist + load-order state files (the new profile starts as a fork of
  the source).
- **Wabbajack-into-new-profile.** Importing a `.wabbajack` while the
  current profile already has mods now offers a 3-way dispatch:
  *Install into NEW profile* (default) / *Replace current modlist* /
  *Cancel*. The new-profile name is suggested from the WJ filename. Test
  the WJ; switch back to your daily driver from the toolbar; nothing
  was overwritten.
- **Clean-modlist mods-dir prompt.** First install in a profile that has
  no mods directory yet pops a prompt with a sensible default
  (`<XDG_DATA_HOME>/nerevarine/<game>/<profile>/mods`) and a "pick a
  different location" escape hatch.
- **Consolidate Mods button.** When a profile's modlist references mods
  that physically live in another profile's mods dir (typical after a
  clone), the Modlist Summary dialog grows a "Consolidate N outside
  mod(s)…" button that moves those folders into the active profile's
  mods dir. Same safety gates as Move Mod Library: free-space check,
  collision check, atomic same-FS rename with cross-FS fallback.
- **In-flight installs survive game/profile switches.** Used to be:
  start an install on FNV, switch to OpenMW for a minute, switch back
  to FNV, find your install silently aborted. The placeholder rows for
  status==2 (installing) mods are now stranded across the switch in a
  per-profile parking lot and restored on the way back. The
  InstallController's pending `extractionSucceeded` signal still lands
  on a live pointer.
- **Migration is invisible.** Existing users get a "Default" profile
  that adopts the legacy `modlist_<gameId>.txt` + `loadorder_<gameId>.txt`
  filenames, so first launch under 0.4 produces zero file moves. Only
  newly-created profiles use the canonical
  `modlist_<gameId>__<profileName>.txt` scheme.

### Pinned games trimmed to OpenMW + FNV

The toolbar's game dropdown now lists only **OpenMW (Morrowind)** and
**Fallout: New Vegas** by default. Other games (Skyrim, Oblivion,
Fallout 4, Cyberpunk, Witcher, Stardew, Gothic, Dark Souls, Mortal
Shell, Skyblivion, Skywind, Fallout London…) are hidden for this
release pending per-game testing of the install + launch paths. Power
users with state from a pre-0.4 install on one of those games flip
**Settings → Show all games (experimental)** to surface the legacy
list — see the matching entry under Performance & reliability for
context.

## Bug fixes

### BSA-shipped mods rendered with `[None]` textures

A user running `ori` on the Wolverine Hall inn sign saw

```
Model: meshes/nigedo/furn_innsign05.nif
Bound textures: [None] [None]
```

while the .esp from `Authentic Signs IT` was loaded fine. The mesh
referenced textures packed inside `Authentic Signs IT.bsa`, but the
writer never emitted a `fallback-archive=<basename>` line for any mod
BSA - only the three vanilla ones (Morrowind / Tribunal / Bloodmoon).
OpenMW does **not** auto-discover BSAs from `data=` directories, so the
archive was invisible to the resource resolver, and every reference
backed by a BSA-packed texture rendered untextured.

Two changes:

- `MainWindow::syncOpenMWConfig` now walks each installed mod's
  `modPath` recursively for `*.bsa` (case-insensitive) and feeds the
  basenames into a new `ConfigMod::bsaFiles`. Recursive because
  Tamriel Data and similar bury BSAs in subfolders alongside their
  plugins.
- `renderOpenMWConfig` collects BSAs from enabled mods, drops any
  pre-existing preamble `fallback-archive=` whose basename is now
  managed (so a renamed/uninstalled mod's stale entry can't survive),
  and emits each managed BSA as `fallback-archive=<basename>` inside
  the `BEGIN/END` block - before the matching `content=` line so the
  archive is registered when OpenMW resolves plugin references.
  Vanilla `Morrowind.bsa` / `Tribunal.bsa` / `Bloodmoon.bsa` keep their
  preamble entry untouched (they're never in any managed mod's
  `bsaFiles`).

Covered by 5 new cases in `tests/test_openmw_config_writer.cpp`:
emit-when-enabled, drop-when-disabled, vanilla preserved, preamble dup
scrubbed, idempotent re-render.

### Completionist Patch Hub: lua bodies still missing after 0.3.1

The 0.3.1 fix added Pass D auto-tick + skipped the empty `scripts/`
placeholder, but a follow-up report showed the lua scripts STILL
weren't on disk after install. The FOMOD's plugins listed each
`Completionist - <X>.omwscripts` as a `<file>` entry but never
declared the matching `scripts/Completionist - <X>/quests_<X>.lua`
bodies (which live in the per-plugin `00 <X>/` subfolder). Ticking
every option only installed the manifests.

`fomod_scripts::installDeclaredScripts` (new) parses every
`.omwscripts` file the wizard copies, extracts each
`<CONTEXT>: <path>` declaration, and pulls the real lua across from
the archive (resolves against archive root first, then the manifest's
own parent directory as a fallback). Existing destinations are left
alone so a well-formed FOMOD's `<folder>` entry isn't overwritten.
Covered by 8 cases in `tests/test_fomod_scripts.cpp`.

### Steam library detection missed custom mounts

`findSteamGameExe` / `findSteamLauncherExe` only scanned three
hardcoded paths, so a user with FNV installed on
`/mnt/nvme_2TB/SteamLibrary` saw "Fallout: New Vegas was not found in
Steam, Heroic, or Lutris." New `steamCommonRoots()` helper parses
Steam's `libraryfolders.vdf` (modern + legacy formats, four candidate
locations including the Flatpak Steam install) and feeds every
configured library root into the scan list.

### FNV launcher hit a Wine error

`onLaunchSteamLauncher` was directly invoking `FalloutNVLauncher.exe`
via `QProcess::startDetached`. Linux can't exec a Windows .exe
naked - the user got a Wine error from whatever fallback association
Linux picked. Steam-installed games now route through
`steam://launch/<appId>` whenever a confirmed local install is found,
so Steam invokes Proton with the correct prefix and runs the game's
default launch option (the launcher for Bethesda titles). Direct
`.exe` invocation only ever runs from the user-picked-an-exe escape
hatch.

### Separator name overlapping the count

Maximised window or just any long separator name ("La Calle de
Cozumel" was the report) drew through the right-aligned `(active/total)`
count because both `drawText` rects spanned essentially the full row
width. The title's rect now reserves horizontal space pre-measured
with the count's font, and the title elides on overflow rather than
clipping mid-glyph.

### Column header separators offset in maximised window

`separatorAt()` walked leftward from `bar.width() - rm` peeling off
column widths, but skipped `wVideoReview` even though Video sits to
the right of Size. Every separator hit-test was off by `wVideoReview`
pixels, very visible at maximum width. The same bug bit `case 0`'s
max-status-width clamp; both now include the video column in the
right-side total.

## Performance & reliability

### UI grey-freeze on Add / Edit mod, Alt+F4 close stall

Adding or editing a mod ran `saveModList`, which chained
`reconcileLoadOrder` + `syncOpenMWConfig` + the missing-master /
dependency / section-count scans. Each of those iterated every
installed mod and called `plugins::collectDataFolders` directly - a
recursive `QDir` walk per mod, twice per save (`reconcile` then
`sync`), plus a separate recursive `QDirIterator` per mod for `*.bsa`
discovery. On a 100-mod modlist that's hundreds of thousands of stat
calls on the UI thread for what was, semantically, a metadata edit.
The window grayed out for several seconds. Same chain runs from
`closeEvent`, so Alt+F4 stalled too.

Three layers of fix:

- **Cache the directory walks.** `reconcileLoadOrder`,
  `syncOpenMWConfig`, and `runMissingMastersScan` now route through
  `m_scans->cachedDataFolders()` instead of calling
  `plugins::collectDataFolders` directly. The session-long cache is
  invalidated for the just-changed path on add / remove (existing
  `invalidateDataFoldersCache(path)` calls), so warm hits short-circuit
  to a hash lookup. New `ScanCoordinator::cachedBsaFiles` does the same
  for the recursive `*.bsa` scan, sharing the same cache key so a
  single `invalidateDataFoldersCache` clears both.
  `warmDataFoldersCache` now pre-warms BSAs in the same worker pass
  it already used for data folders.
- **Move file IO off the UI thread.** `saveModList`'s modlist snapshot
  backup + write, and `syncOpenMWConfig`'s openmw.cfg + launcher.cfg
  snapshot-backup / write / mtime-gate / render / write chain, are
  handed to `QtConcurrent::run` and tracked in a single
  `MainWindow::m_lastSaveFuture`. Each new submission first
  `waitForFinished()`s any in-flight prior write, so writes are
  serialized - no two workers can race on the same file. Render of
  openmw.cfg + launcher.cfg inputs stays on the UI thread (it consumes
  cached UI state); only the disk hits move.
- **Flush on close.** `closeEvent` waits on `m_lastSaveFuture` after
  scheduling the final save so Alt+F4 still flushes user state to disk
  before the process exits, and now also stops the conflict / masters
  debounce timers up-front so they can't fire during teardown.

Per-stage timings now land in `log.txt` so any single number jumping
in a future release is bisectable:

```
saveModList ms total=… absorb=… writeQueue=… loadorder=… syncCfg=… masters=… deps=… counts=… items=N
syncOpenMWConfig ms total=… scanMods=… readCfg=… masters=… render=… writeQueue=… mods=N
addModFromPath ms=… name='…'
closeEvent ms total=… save=… flush=…
```

### Async-write failures surface as a red banner

The new `QtConcurrent::run` chain in `saveModList` /
`syncOpenMWConfig` previously silenced any `QFile::open` failure on
the worker - read-only mount, full disk, AppImage writing into the
wrong path - all logged a `qCWarning` to log.txt that the user never
saw.  Workers now route the failure through
`QMetaObject::invokeMethod(this, …, Qt::QueuedConnection)` to a UI-
thread slot that flashes a 7-second NotifyBanner naming the file
that failed, plus an 8-second status-bar fallback.  `QPointer<MainWindow>`
in the lambda guards against the window being destroyed mid-write.

### Debounced save for rapid mutations

Hold-Alt-↑ to move a mod up the list, drag-drop a multi-row
selection, or rapid-toggle the favorite star: each of those used to
fire its own `saveModList` synchronously, and the
`m_lastSaveFuture.waitForFinished()` serialization at the top of
each save chained them N×.  Routed five callers (drag-drop
`rowsMoved`, `onMoveUp` / `onMoveDown`, `favoriteToggleClicked`,
`separatorCollapseToggleClicked`) through a new
`scheduleSaveModList()` that restarts a 150ms single-shot QTimer.
Multiple calls within the window collapse into one save.
`saveModList()` itself stops the timer at entry, so any synchronous
caller (closeEvent, profile / game switch, undo apply, bulk install
completion) can't be raced by a stale schedule.  closeEvent also
explicitly stops the timer up-front as belt-and-braces.

### TES3 master cache

`reconcileLoadOrder`'s topological pass used to call
`plugins::readTes3Masters` once per plugin per save - cheap (~1 KB
read each) but on a 100-plugin order that's 100 file opens every
saveModList.  ScanCoordinator gained a `(path, mtime)` cache that
zeroes it out for warm hits; mtime miss re-reads, file deletion
drops the entry, and `invalidateDataFoldersCache(modPath)` sweeps
file-level entries under that mod root so uninstalls don't leak.
Six new assertions in `tests/test_scan_coordinator.cpp` pin
`(path, mtime)` keying, post-overwrite re-read, and prefix-sweep
invalidation against hand-built TES3 fixtures.

### Install-pipeline naming heuristics carved out

The folder-name heuristics that
`MainWindow::addModFromPath` ran inline (sibling-folder dedup,
generic-name detection, trailing-version-chain strip, hard-coded
overrides) moved to a new `mod_naming` namespace.  MainWindow keeps
the QListWidget mutation; the regex thresholds and "what counts as
a Nexus archive slug" decisions are now in pure free functions
unit-testable without any modlist or filesystem state.  36
assertions in `tests/test_mod_naming.cpp` pin the four entry
points: `findStaleSiblings`, `folderNameLooksGeneric`,
`stripTrailingVersionChain`, `hardcodedRename`.

### saveModList file write extracted to modlist_io

The `QtConcurrent::run` worker in `MainWindow::saveModList` used
to inline the snapshot-backup + open + write + error-extraction
sequence.  Carved out as `modlist_io::writeModlistFile(path,
content) -> std::optional<QString>` (nullopt = success, value =
QFile::errorString of the open failure).  The async-write banner
plumbing depends on this returning the error rather than
swallowing it; `tests/test_modlist_io.cpp` (16 assertions) now
exercises the success path, snapshot-backup creation, read-only
directory, empty path, empty content, sequential overwrites, and
1 MB roundtrip - the highest-risk uncovered code in the new
async chain has end-to-end coverage.

### syncOpenMWConfig pure orchestration carved out

The 600-line orphan-rescue + launcher-augmentation + master-
satisfaction + scrub block inside `MainWindow::syncOpenMWConfig`
is now a free function `openmw::prepareForSync` in
`src/openmwconfigwriter.cpp`.  MainWindow keeps the m_modList
snapshot phase (UI-thread-bound) and the m_loadOrder mutation +
async write delegate; everything between is pure on top of
QtCore filesystem access and unit-testable through the
`SyncPrepareInputs` / `SyncPrepareResult` value types.  No
behaviour change - the historical bug-report block comments
moved verbatim into the new function.  9 scenarios pinned in
new `tests/test_openmw_sync_prep.cpp` (24 assertions): orphan
content scrubbed, orphan data= path scrubbed, orphan-managed
rescue, launcher-only externals augmented, managed-path
synthetic suppressed, unsatisfied-master suppression,
satisfied-master preservation, groundcover dropped from load
order, empty inputs.  This is the first unit-level coverage of
the orchestration layer.

### AppImage env-leak health probe

Memory log notes recurring "QIBusPlatformInputContext: invalid
portal bus" failures when LOOT or openmw-launcher inherit our
bundled Qt env (`LD_LIBRARY_PATH` / `QT_PLUGIN_PATH` /
`LD_PRELOAD` pointing into the squashfs).  `childProcessEnvironment()`
restores `_ORIG` values before launching foreign Qt apps, but a
new env var that wasn't captured by linuxdeploy can silently
leak.  `main.cpp` now walks the same scrub list at startup and
emits one `qCWarning` per leak so future regressions surface in
log.txt of the very first session, plus an `qCInfo` "AppImage
env probe: clean" when nothing is suspect.  No-op outside the
AppImage runtime.

### Pinned games dropdown gets an experimental "show all" toggle

The 0.4 game dropdown ships pinned to OpenMW + FNV.  Skyrim /
Oblivion / Fallout 4 / etc. are commented out pending per-game
testing of the install + launch paths.  New users with state
from a pre-0.4 install on one of those games hit a wall today;
new Settings → "Show all games (experimental)" preference (off
by default) un-hides the legacy game list so they can switch
back without editing QSettings.  Mirrors the
`launch/skip_reboot_check` precedent.

### First-run wizard probe under AppImage

`maybeShowFirstRunWizard` falls back to a disk probe for
`modlist_<gameId>.txt` when `m_modList` is empty (covers the case
where the user's modlist file hasn't been loaded for the active
profile yet).  That probe used to hardcode `applicationDirPath()`,
which is the read-only squashfs mount under AppImage - so an
AppImage user with state in `AppDataLocation` would silently fail
the probe.  Now routed through `resolveUserStatePath` (the same
helper `loadModList` uses), with the binary-adjacent paths kept as a
legacy fallback for older dev/release builds.

## Migration notes

- Existing modlist + load-order files are NOT renamed. The auto-created
  "Default" profile adopts them in place.
- A pre-existing `fallback-archive=<mod_basename>` line in the preamble
  that names a BSA Nerevarine now manages will be moved into the
  managed section on next sync. Vanilla BSAs are never touched.
- If you previously added a non-OpenMW / non-FNV game profile (Skyrim
  etc.), it stays in QSettings but is hidden from the dropdown. Re-show
  via **Settings → Show all games (experimental)**.
