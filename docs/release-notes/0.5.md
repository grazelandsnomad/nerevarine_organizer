# 0.5 release notes

A user-requested **Dark mode**, three FOMOD installer fixes — **no more
case-variant duplicate folders** (the Project Atlas "`Meshes` *and* `meshes`"
report), **optional radio-button steps you can actually decline**
(OAAB_Saplings), and **installers buried in a wrapper folder are now detected**
(Completionist Patch Hub) — on top of two internal consolidations: a
**`ui::` prompt-helper layer** and a single **`subprocess::` chokepoint** for
launching external programs.

## New features

### Dark mode

A **Dark mode** / **Light mode** toggle now sits at the right end of the
toolbar. Click it to switch the whole application between the native light
theme and a dark theme; the button always names the theme it will switch *to*.
The choice persists across restarts and is applied before the window is built,
so there's no light-to-dark flash on launch. Defaults to light for existing
users.

Implemented as an application-wide palette swap (dark = Fusion style + a dark
`QPalette`; light = the captured native style + palette) so it covers every
standard widget — lists, menus, dialogs, inputs — rather than a per-widget
stylesheet. New `theme::` module + `Settings::uiDarkMode()` (`ui/dark_mode`).

**Separators follow the theme.** Separator colours are authored for a light
list, so in dark mode every separator's background is darkened toward black
*while keeping its hue* — a pink section stays pink, a blue one stays blue,
just deep enough to belong in a dark UI — with light text on top for
readability. This is render-only: your saved colours are never modified, so
switching back to light restores the exact original look and nothing in your
modlist changes.

## Bug fixes

### FOMOD installs created duplicate `Meshes` / `meshes` folders

Installing a FOMOD mod and picking patches could leave two directories
that differ only in case — e.g. a `Meshes` *and* a `meshes` — with no
clear winner and no easy way to resolve them by hand. Project Atlas was
the report.

FOMOD `ModuleConfig.xml` files are authored on Windows, where the
filesystem is case-insensitive: `Meshes\` and `meshes\` are the same
folder. A mod's base option and its patch options routinely spell that
folder differently, and the installer was creating each destination
directory with whatever casing the option happened to use. On a
case-sensitive Linux filesystem that forks two real directories; OpenMW's
VFS folds case and pulls assets from both, so which copy "wins" is
ambiguous — exactly why it was confusing and tedious to fix by hand.

The install now emulates the case-insensitive filesystem the mod was
authored for:

- New `fomod::resolveDest(root, rel)` — the write-side twin of the
  existing `fomod::resolvePath` (which resolves *source* paths and returns
  empty on a miss). `resolveDest` walks each destination component,
  reuses the on-disk casing of any component that already exists
  (case-insensitive match), and adopts the authored casing only for
  genuinely-new components. It never fails. So the second option's
  `meshes` lands inside the first option's already-staged `Meshes`
  instead of forking a sibling.
- Every FOMOD destination now routes through it:
  `FomodWizard::applySelections` (both the folder and file branches) and
  the recursive `fomod_copy::copyContents` / `copyDir` helpers, instead of
  raw `dstDir + "/" + name` concatenation.
- **Last writer wins.** File copies now `QFile::remove` the target before
  `QFile::copy`. `QFile::copy` refuses to overwrite, so once two options
  merge into one folder a later patch could otherwise be silently blocked
  by the base file it was meant to replace. Removing first lets the
  later-installed option (the patch) take effect — again matching the
  Windows behaviour.

Covered by a new "case-variant folders merge into one" scenario in
`tests/test_fomod_copy.cpp` (no-duplicate-folder, merged-casing,
both-files-present, last-writer-wins) plus three `resolveDest` cases in
`tests/test_fomod_path.cpp`.

This only affects **new** installs — a mod already on disk with duplicate
`Meshes` / `meshes` folders is not merged retroactively; reinstall it (or
merge the two folders by hand) to clean it up.

### FOMOD radio-button steps with no way to decline

OAAB_Saplings (steps 1 and 3, among others) presented radio-button option
groups with no way to pick *none* — you were forced to select a patch even
when you wanted none of them.

FOMOD groups carry a selection type. `SelectExactlyOne` genuinely requires a
choice (resolution, texture size, …) and stays mandatory. But
`SelectAtMostOne` means *zero or one*, and the installer was treating it
identically to `SelectExactlyOne`: rendering an exclusive radio group and
force-checking the first option, with no control that could return the group
to "nothing selected".

`SelectAtMostOne` groups now:

- no longer force a default selection — they start on "none" unless a plugin
  is Required / Recommended or matched by the smart-default passes, and
- gain an explicit **None — install none of these** radio in the same
  exclusive group, so picking a plugin clears None and picking None clears the
  plugins.

`SelectExactlyOne` groups are unchanged — a choice there is still required, as
the FOMOD intends.

### FOMOD installer not detected inside a wrapper folder

Some archives — Completionist Patch Hub is the canonical case — keep
`fomod/ModuleConfig.xml` one or more levels below the mod root, typically inside
a Nexus wrapper folder. When a stray file sits beside that wrapper at the top
level, the post-extraction "dive into the single subdirectory" heuristic
correctly refuses to descend (diving would orphan the top-level file), so the
FOMOD ended up below where detection looked. The result: **no wizard at all** —
the archive was installed as a plain copy, so none of the FOMOD logic ran,
including the rescue that pulls each patch's `.lua` body out of its per-plugin
folder. The visible symptom was the `.omwscripts` manifests landing at the mod
root with an empty `scripts/` beside them, and OpenMW then failing to load the
scripts those manifests declare.

Detection used to scan only the immediate children for `fomod/ModuleConfig.xml`.
New `FomodWizard::findFomodRoot` does a bounded breadth-first search for the
shallowest directory that directly contains it; `hasFomod` uses that, and the
wizard rebases its archive root onto the discovered folder so source paths
resolve and the staging dir lands correctly. Wrapped and nested FOMODs now open
the wizard as expected, and a multi-FOMOD pack picks the topmost installer.

Covered by four `findFomodRoot` cases in `tests/test_fomod_wizard_ui.cpp`
(direct, wrapper-with-sibling-file, none-anywhere, shallowest-wins). Verified
end-to-end against Completionist Patch Hub (112 manifests): every patch's
`.lua` installs under `scripts/<patch>/` and the in-game quest tracker lists
the patched categories.

## Reliability & maintainability

### `ui::` prompt helpers

~80 raw `QMessageBox` call sites collapsed onto a small
`ui::confirm` / `ui::warn` / `ui::info` / `ui::critical` helper layer, and
the remaining three-argument `question()` sites converted to
`ui::confirm`. Simple yes/no and notice dialogs now go through one
consistent entry point; complex multi-button boxes stay raw. No
user-visible behaviour change — it removes a large class of
copy-paste divergence around button roles and default buttons.

### `subprocess::` — one chokepoint for launching external programs

Every external-process launch (LOOT, the OpenMW Launcher, the game, file
managers, …) now goes through a single `subprocess::` namespace. The
practical payoff: the AppImage Qt environment scrub —
`childProcessEnvironment()`, which restores the `_ORIG` values of
`LD_LIBRARY_PATH` / `QT_PLUGIN_PATH` / `LD_PRELOAD` before handing control
to a foreign Qt child — is now applied **everywhere**, not just at the
launch sites that remembered to call it. Foreign Qt apps inheriting our
bundled-Qt environment was the cause of the recurring
`QIBusPlatformInputContext: invalid portal bus` silent exits when running
from the AppImage.

### Portable Linux builds — no more glibc breakage on Fedora / Steam Deck

Releases built on a bleeding-edge host (glibc 2.43+) were failing on Fedora 42
and similar with `version 'GLIBC_2.43' not found`, for both the AppImage and the
plain build. Two changes:

- `build-appimage.sh` now **aborts before packaging** if any bundled library
  still requires a glibc newer than 2.41 (the patch target), so a too-new build
  can never be shipped as a hotfix again.
- New `packaging/Dockerfile.portable` + `build-portable.sh` build both the
  AppImage and the plain release **inside an old-glibc container** (Ubuntu
  24.04, glibc 2.39), so every artifact targets an old glibc natively and runs
  on Fedora, Steam Deck and every newer distro — the durable fix.

## Test count

33 unit tests, all green. New this release: `test_fomod_wizard_ui` — drives
`FomodWizard::buildUi` headless (offscreen QPA) to cover the `SelectAtMostOne`
"None" radio, the smart-default passes, and the recursive `findFomodRoot`
wrapper-folder detection. The case-merge / `resolveDest` / path-traversal
guards extend `test_fomod_copy` and `test_fomod_path`.

## Migration notes

- No state-format changes. Existing modlists, load orders and profiles
  load unchanged.
- The FOMOD case-folder fix applies to new installs only; pre-existing
  duplicate `Meshes` / `meshes` folders are left as-is — reinstall the
  affected mod to collapse them.
- Reinstalling a mod whose FOMOD has `SelectAtMostOne` steps may now default
  to "none" where it previously pre-selected the first option; your own saved
  choices are still restored.
